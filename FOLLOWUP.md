### 6. 补充一个真实例子

请结合这次作业，或结合你过去的真实经历，讲一个你主动负责、推进问题闭环、在压力下保证结果的例子。重点说明你当时怎么判断、怎么沟通、怎么推进、最后结果如何。
我的经历：云南创新遴选项目
当时我负责云南创新遴选项目，在课题评选答辩前仅剩一周时，发现大模型存在两个致命问题：回答准确率低（幻觉严重），且部分返回时长过长，几乎无法在答辩中做稳定演示。如果带着这个问题上场，不仅会丢分，甚至可能直接被淘汰。
1.我怎么判断
我首先做了快速的风险评估：
剩余时间只有7天，重新训练或换模型路线完全不现实。
答辩的核心环节是评委现场提问，模型必须能准确抽取MCP（Multi-Channel Platform）中的真实数据，并给出流畅回答。
两个问题中，回答时长更容易通过工程手段优化，幻觉问题更棘手，但可以通过提示词约束和数据预处理降低。
优先级：保证重点问题的回答“不卡壳、不错得离谱”，而不是追求完美。
2.我怎么沟通
第一时间找项目负责人：不是等开会，而是直接面对面说明风险和我的初步判断，明确告诉他“我需要人力支持，但不会耽误答辩”。
争取关键资源：请求协调1个人，专门帮我提前准备好答辩可能问到的测试用例数据（包括真实MCP中的典型场景和边界案例），确保我们在优化时有明确的目标。
对齐团队预期：召集相关同事，坦诚说明现状：“目前模型不可用，但如果我们接下来3天集中攻关，有把握达到演示底线。” 把大家从焦虑拉回行动。
3.我怎么推进
分两条线并行：
工程优化线：我盯着MCP数据抽取接口，发现耗时主要来自多次网络调用和未做缓存。我协调后端同学加上本地缓存和并行请求，将平均返回时长从6秒压到了1.5秒以内。
幻觉压制线：利用那1位同事提前准备好的测试用例，我和算法同学反复迭代提示词——加入“必须优先使用MCP返回的真实数据”、“如果数据缺失，回答‘暂未查询到相关信息’，严禁编造”等强约束。同时增加一个简单的输出后处理，匹配不到数据时强制走标准回答模板。
连续加班但控制节奏：前三天每天晚10点下班，第三天晚上跑通完整演示链路。我要求团队第四天恢复正常节奏，保证答辩当天精力充沛。
模拟答辩演练：第五、第六天用准备好的测试用例进行两轮完整模拟，让项目负责人扮演评委随机提问，我操作模型演示。每次发现问题立刻记下来，当晚修复。
4.最后结果如何
答辩当天，模型对评委提出的所有问题都能流畅（1-2秒内响应）且准确回答，没有出现一次幻觉或卡顿。
我们最终在行业评选中获得第一名，客户和评委对系统的智能化水平给予高度评价。
事后复盘：我们把这次紧急优化的提示词模板固化下来，并建立了“提前至少2周进行压力测试与幻觉检测”的内部流程，避免再次踩坑。


### 7. 如果文档章节层级解析不对

假设这套系统解析后发现不少文档的章节层级不准确，例如标题、条款、段落、表格归属或跨页内容关系识别错，导致后续检索和问答效果受影响。

请说明你会如何定位问题、判断影响范围、设计修复方案，并验证修复是否有效。也可以说明短期交付时如何止损，长期如何把这类问题做得更稳定。

定位 → 评估 → 修复 → 验证（摘要流程）

定位问题（快速可复现）
复现样本：收集 20 个出问题的 PDF（含数字/扫描/复杂表格/跨页条款）。
重放解析流水线：对每个样本运行 parse_pdf() → 查看 PageContent.text 与 tables 输出；运行 build_chunks() → 检查 Chunk.page、clause_id、text 切分边界。查异常样例并保存原始解析产物（文本、blocks、表格、OCR 输出）。
检查解析元数据：在 pdf_analyzer.py 用 page.get_text("dict") 的 blocks、bbox、font 信息定位标题/段落断裂处；在 pdfplumber 的 table bbox 检查跨页表格是否被截断。
日志/可视化：把每页的文本与块 bbox、识别到的 clause 起始位置导出为可阅读文件（HTML 或简单可视化）以人工快速定位跨页归属问题。
判断影响范围（定量/定性）
统计指标：计算样本集中以下比率：无 clause_id 的 chunks 比例；跨页条款被错误拆分次数；表格被截断或表格行丢失率。
检索影响：在问题集上运行检索基线，比较“使用原索引”与“理想（人工修正）索引”间 Top‑3 命中率差值，量化性能下降对业务的影响。
风险分类：把问题归类为“高影响（导致错误答复/误导）”、“中影响（降低质量）”、“低影响（仅表现为表现损失）”。
短期止损方案（可在演示/试用期快速交付）
增强重叠与回退合并：
在 chunker.build_chunks() 增加页面间合并策略：若某 page 的首 chunk 无 clause_id，尝试合并前后 page 的 chunk（扩大 overlap）。
对表格：若 pdfplumber 提取的 table bbox 接近页边界，自动尝试合并相邻页的 table 文本片段。
标注低置信并人工复核：为每个 Chunk 添加 confidence（基于是否检测到 clause header、是否跨页截断等），低置信答案自动标注“请人工确认/拒答”。
临时 UI/体验：在前端暴露原始上下文片段与整页视图，便于人工快速定位与复核。
回滚索引：保存当前良好索引快照，演示时可回退到最近一次“通过”的索引文件。
中长期修复设计（更稳定、系统化）
规则+布局结合：
利用 PDF 字体/字号/样式与 bbox（via PyMuPDF）识别标题与条款头（font size/bold/spacing），把 CLAUSE_RE 与版式线索联合判定 clause 边界。
用 reading-order（块序）和 bbox 连续性判定段落跨页延续。
表格跨页重建：
基于 table bbox 与列数/列对齐度在相邻页寻找候选续表，合并行并保持单元格对齐；对合并失败情形保留原表并标注“可能不完整”。
小型 ML/学习型组件：
收集并标注若干示例（标题/条款/段落/表格续页标签），训练轻量模型（如 LayoutLM / XLM-R + bbox features 或一个随机森林基于样式特征）来判定“块类型”（heading/paragraph/table/continuation）。
先做弱监督：用规则生成伪标签，再人工校验少量样本以提升精度。
可插拔管道与回退：
把解析分成阶段化组件：layout_parser（bbox+font）→ structure_resolver（合并/修正）→ chunker。若新组件失败，可回退老逻辑。
增加断言与监控：
在构建索引时运行结构健康检查（例如“每个文档条款覆盖度不小于 X”），否则标记该文档为需人工处理。
验证修复（自动化与人工）
单元 + 集成测试：为常见失败模式写入 tests（如：跨页标题、跨页表格、被误当作正文的页眉），确保修复后回归测试通过。
回归对比指标：在样本集上比对修复前/后指标：Top‑3 命中率、无 clause_id 比率、表格完整率。目标为显著改善检索命中与降低低置信答案比率。
人工抽样验证：修复上线后抽查 100 条回答/证据对，要求领域专家确认溯源正确率达到预定阈值。
Canary 灰度：先对 10% 文档或小比例请求启用新解析器，监测错误率与业务指标，再放量。
具体实施步骤（建议的短期行动项）
步骤 A（今天/24h）：收集 20 个失败样本并导出解析产物；实现 chunker 的 overlap 增强与低置信标注；添加快速烟雾测试。
步骤 B（1–3 天）：实现表格跨页合并 heuristics；在 parser.py 中导出 block bbox/font 信息供判断；在前端显示原始页片段。
步骤 C（3–7 天）：开发 layout_parser 原型（基于 bbox+font rules），编写回归测试；部署 canary。
步骤 D（2–4 周）：标注训练数据、训练/评估 ML 结构解析器、把模型加入管道并用更多回归用例验证。
指标/验收（修复生效的判据）
目标量化例子：无 clause_id 的 chunks 比率下降 ≥ 60%；Top‑3 命中率提升 ≥ 10%；跨页表格完整率提升 ≥ 70%；低置信答案率下降 ≥ 50%。
验收流程：自动指标通过 → canary 观察 48h 无退化 → 人工抽样审查通过 → 全量启用。

### 8. 如果系统效果不稳定

请结合你提交的系统，选一个你认为最可能影响效果的问题，说明你会如何定义评价标准、收集 bad case、定位原因、尝试优化，并判断优化是否真的有效。

主要问题: 检索证据不可靠或缺乏可溯源的 page/clause 信息，导致 LLM 生成的答案“幻觉”或无法证明每条断言的来源。

评价标准
Top‑k 命中率: 在标注问题集上，Top‑3 中包含正确证据的比例（目标 ≥ 80%）。
证据覆盖率: 答案中每个断言被证据支持的覆盖比例（句级对齐，目标 ≥ 90%）。
拒答/误接受率: 无相关证据时系统应拒答，误接受率 < 5%。
溯源可用性: 返回的 Evidence 必须包含 page（和可选 clause_id），缺失率 < 2%。
平均延迟: 检索时间（不含 LLM）与总体响应时间满足 SLA（例如 <3s / <10s）。
用户可视化校验通过率: 人工抽样中能被领域专家判定为“溯源正确”的比例（目标 ≥ 95%）。

如何收集Bad Case
自动化收集: 在测试集与线上 query 上记录所有未通过自检 (self_check() 失败)、无 Evidence 或 Evidence.page 缺失的请求，保存原始 PDF、查询、返回证据与 LLM 输出到 uploads/badcases/。
人工标注用例集: 从客户/产品处收集典型问题并人工标注“正确证据位置”（页/段）；把失败样例标为高优先级。
对比基线: 把当前索引结果与人工金标准做差异对比，筛选 Top‑k 中无正确证据或页码错误的样本作为 bad case。
用户反馈通道: 在前端增加“证据错误/不可信”按钮，把用户上报的样本也收集进 bad case 池。

定位原因（诊断流程）
复现样例: 取若干 bad case，重放解析管线：parse_pdf() → chunker.build_chunks() → TfidfStore.search()（或 embedding store）。导出中间产物（page text, blocks, chunk boundaries, index entries）。
检查解析质量: 验证 PDF 解析是否丢失文本、页码是否丢失、表格/跨页内容是否被截断（查看 pdf_analyzer.py 的 block/bbox 输出）。
检查切片/索引: 验证 chunk 的 page/clause_id 是否正确写入索引；检查 overlap 大小、chunk 长度、是否有太多短/空 chunk 导致检索偏移。
检索器问题: 对 TF‑IDF 检索结果检查权重分布、倒排索引覆盖率、停用词处理与分词（中文分词）是否合适。
OCR 问题: 对扫描件确认 OCR 输出文本错误率、页码对应是否错位。
LLM 对齐问题: 通过把 LLM 输出句子逐句与返回的 Evidence 做二次验证（用 LLM 或规则），定位哪些断言没有证据支撑。

尝试的优化（工程 + 算法）

短期（低成本）
在 chunker.build_chunks() 增加更大 overlap 与跨页合并 heuristics，确保 clause 头尾不被切断。
强化索引元数据：确保每个 chunk 在索引中包含 page 和 clause_id 并在检索结果中返回。
提升自检 self_check()：把证据覆盖度、证据数量与置信阈值合并判定，低置信直接拒答或要求人工复核。
在前端展示原始片段与页码，便于人工快速定位并标注 bad case。
中期（质量改进）
引入语义向量检索（embeddings + FAISS/Chroma）作为主要检索，保留 TF‑IDF 作为回退。
用小型验证 LLM 步骤做“事实-证据对齐”：对每个 LLM 生成的断言，调用校验器确认是否能被某条 evidence 支撑（若否，标注并拒答或在答案中明确标记“未经证据支持”）。
针对 OCR 错误高的文档开启更稳健的 OCR 或人工校正流程。
长期（稳健化）
在解析阶段（parser.py / pdf_analyzer.py）加入基于 bbox/font 的 heading detection 与表格跨页重建，或训练轻量 Layout 模型（LayoutLM 微调）来识别 clause 边界。
建立监控与报警：Top‑k 命中率、证据缺失率等指标异常自动告警并触发回退策略（例如切换为只返回原文片段）。

如何判断优化是否真正有效

量化对比: 在保留的标注测试集上跑 A/B（旧版本 vs 新版本），比较核心指标（Top‑3 命中率、证据覆盖率、误接受率、延迟），并计算置信区间与显著性（例如 McNemar 或 bootstrap）。
回归与回放测试: 对采集的 bad case 池逐条回放，统计被修复的比例（目标显著下降）。
人工抽样审核: 随机抽样 N=100 条输出，领域专家判定溯源正确率；要求通过率达到预设阈值。
Canary 发布: 在小流量（如 10%）上先上线改进；观察 48–72 小时无退化再放量。
线上用户反馈率: 监测“证据错误”举报率下降与用户满意度提升。

## 1. 遇到不清楚但又比较急的需求

如果产品或客户给了一个比较模糊的目标，但希望尽快看到结果，你通常会怎么处理？

请说明你会先确认什么、如何推进第一版、如何避免方向做偏。

## 2. 关键节点前发现问题

如果周五晚上发现系统还有一个影响演示或交付的问题，但周一就要给客户看，你会怎么安排？
请说明你会如何判断优先级、同步风险、协调资源，以及你自己会投入到什么程度。

1.判断优先级：P0阻断性问题（核心流程不通）→通宵修复；P1观感问题→找绕过方案或现场解释；P2边缘问题→记录并避开。
2.同步风险：当晚拉会明确问题和修复方案，同步给客户/老板——周五告知风险，而非周一才暴露。
3.协调资源：我+1名核心开发连夜定位，分段承诺（先试2小时），团队轮休不熬干。我全程在线，必要时亲自写脚本做兜底。
4.个人投入：我会睡在办公室直到问题解决或找到可接受替代方案。但如果周日仍不稳定，主动建议切换备选演示（录屏/PPT），不隐瞒、不赌运气。

## 3. 沟通中出现分歧

如果产品、客户或技术同事对同一个问题的理解不一致，甚至有人比较着急，你会怎么沟通？
请说明你如何保持推进节奏，同时避免沟通失控或反复拉扯。

1.快速判断分歧类型（目标/信息/情绪），拉15分钟短会，不群里争论。
2.用“事实漏斗”对齐：复现问题→列出各方假设→锁定可验证的一个事实→指定决策人。
3.分两条线推进：行动线（定最小下一步、明确责任人和时间）；情绪线（单独接住着急的人，承诺兜底）。
4.避免拉扯：时间盒15分钟、书面确认结论、区分“决策权”与“建议权”。

## 4. 多件事情同时压过来

3.每天固定时间（如午休前、下班前）：整合进度，同步团队，重新调整优先级。

## 5. 讲一个真实例子

请讲一个你过去主动负责、推进问题闭环，或者在压力下保证结果的例子。
重点说清楚：当时是什么情况、你怎么判断、怎么沟通、怎么推进、最后结果如何。
请尽量真实具体。我们不期待“完美答案”，更想看你处理事情的方式。

当时是什么情况
我负责反诈公共服务平台从外网迁移至内网的工作。这是一个涉及多部门协同的关键项目：平台原本部署在外网，因安全要求必须整体迁入内网环境，并最终实现内外网双主策略（同时提供服务，互为备份）。
迁移过程中出现了一个严重问题：外网用户访问平台变得极其缓慢，部分页面加载超过10秒，几乎不可用。而当时已经向客户承诺了迁移时间表，且反诈业务不能中断。

我怎么判断
我首先快速定位问题的责任边界——因为涉及天翼云（外网入口）、专线（跨网传输）、信通院（内网安全策略）、运维部门（网络配置），任何一个环节出问题都会导致访问慢。但各部门往往只看到自己的那一部分。

我的判断是：
这不是单一技术故障，而是跨部门网络链路配合问题。单纯在自己的平台侧优化代码解决不了。
必须主动拉起各方进行联合排查，否则会陷入“互相推诿、原地打转”的局面。
优先保证核心业务（反诈查询）不彻底中断，哪怕临时采用降级方案。

我怎么沟通
对内（上级领导）：第一时间如实汇报现状，明确说“目前网络链路存在瓶颈，我需要授权去协调天翼云和运维部门做联合调测”，争取到跨部门调度的支持。
对外（客户/信通院）：主动同步风险，但给出台阶——“我们正在做网络深度优化，预计XX小时内恢复访问速度，期间核心查询功能可用”，不隐瞒但也不让客户恐慌。
对平级（天翼云、运维）：不指责，而是带着数据说话。我整理了访问慢的时段、网络延迟、丢包率等初步数据，拉了一个联合排查群，提议：“我们各出一人，用明天上午2小时做一次全链路压测，看瓶颈到底在哪儿。”

验证并固化：完成调测后，我用真实业务场景重新跑了一遍压测，外网访问时长从10秒降到2秒以内。然后与团队一起将优化后的网络配置文档化，并推动内外网双主策略的最终上线。
